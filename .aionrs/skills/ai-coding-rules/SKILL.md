# AI Coding Rules

设定 AI 开发过程中的三类核心规则：代码规范、行为规范和代码范围约束。规范条目根据约束力强弱依次分为 **【强制】**、**【推荐】**、**【参考】** 三大类。

## 规则体系

三类规则按优先级排列（高到低）：

1. **代码范围** — 最高优先级，定义可以/不可以修改的代码边界及项目架构约束
2. **行为规范** — 定义 AI 的开发行为模式（确认策略、异常处理、日志、测试）
3. **代码规范** — 定义代码本身的书写标准（命名、格式、OOP、控制语句、集合、注释）

违反高优先级规则的后果比低优先级更严重。代码范围的违规可能导致系统级故障。

## 1. 代码范围（Code Scope）

### 核心原则

**默认最小修改原则**：AI 只修改与当前任务直接相关的代码。任何超出任务范围的修改都需要明确的用户授权。

### 修改范围定义

| 级别 | 范围 | 说明 |
| --- | --- | --- |
| `strict` | 仅当前函数/方法 | 不允许修改函数签名、上下文以外的代码 |
| `file` | 当前文件 | 可修改当前文件内的任意位置 |
| `module` | 当前模块/目录 | 可修改同一模块下的多个文件 |
| `project` | 整个项目 | 可修改项目级配置、基础设施代码 |

### 禁止修改的代码区域

以下区域默认禁止修改（需用户显式确认后放行）：

- **数据库迁移脚本** — 已执行的 migration 文件不可回改
- **第三方依赖配置** — `*.csproj`、`package.json`、`requirements.txt` 等依赖锁定文件
- **基础设施配置** — Dockerfiles、CI/CD 配置、部署脚本
- **公共 API 接口** — 已发布的 API 签名、接口契约（修改需用扩展方法兼容）
- **身份认证/授权逻辑** — 登录、权限检查、Token 管理相关代码
- **加密/安全相关代码** — 加解密、哈希、签名逻辑

### 新增代码允许范围

- 新建文件/模块：默认允许，但需告知用户
- 新增依赖：需确认后执行
- 新增配置文件：需描述用途后获得确认

## 2. 行为规范（Behavior Conventions）

### 修改前必读原则

**编辑任何文件前，必须先读取其当前内容。** 这确保 AI 基于最新状态进行修改（而非记忆或假设）。

### 确认策略

| 操作类型 | 策略 | 说明 |
| --- | --- | --- |
| 读取文件 | 自动执行 | 不触发确认 |
| 搜索代码 | 自动执行 | Grep / Glob 类操作 |
| 创建新文件 | 告知后执行 | 告知文件名和用途后即可创建 |
| 修改现有文件 | 告知变更摘要后执行 | 说明改什么、为什么改，然后执行 |
| 删除文件/代码 | 必须确认 | 删除不可逆，需用户明确同意 |
| 运行命令 | 告知后执行 | 说明命令目的和执行环境 |
| 安装依赖 | 必须确认 | 影响项目环境 |
| 修改配置 | 必须确认 | 影响运行时行为 |

### 变更说明格式

每次修改需包含：

1. **变更位置** — 文件路径 + 函数/区域
2. **变更内容** — 具体改了什么
3. **变更原因** — 为什么这样改
4. **影响范围** — 此次修改影响的其他区域（可选）

### 不可逆操作保护

- 删除操作前必须备份（git stash 或提示用户先 commit）
- 批量修改（跨多个文件）前必须告知影响范围
- 数据库操作（ALTER/DROP/TRUNCATE）必须二次确认

### 错误处理策略

| 场景 | 行为 |
| --- | --- |
| 命令执行失败 | 分析错误输出，尝试修复后重试（最多 2 次），第 3 次失败向用户报告 |
| 文件不存在 | 确认路径是否正确，不假设文件位置 |
| 权限不足 | 向用户报告所需权限，不尝试提权 |
| 网络超时 | 重试 1 次，仍失败则报告 |

### 异常处理原则（语言通用）

**【强制】** 异常不要用来做流程控制、条件控制。

**【强制】** 捕获异常是为了处理它，不要捕获了却什么都不处理。如果不想处理，请将异常抛给调用者。

**【强制】** 对大段代码进行 try-catch 是不负责任的表现。请区分稳定代码和非稳定代码，对非稳定代码尽可能区分异常类型再做对应的异常处理。

**【强制】** 不能直接使用 catch 来对可通过常规检查避免的异常进行捕获（如 NullReferenceException、InvalidOperationException 等）。应先用条件判断防御。

**【强制】** finally 块必须对资源对象、流对象进行关闭，有异常也要做 try-catch。

**【强制】** 不能在 finally 块中使用 return，finally 中的 return 返回后方法结束执行，不会再执行 try 块中的 return 语句。

**【推荐】** 方法返回值可以为 null，但必须添加注释充分说明什么情况下会返回 null。调用方需要进行 null 判断（使用 `?.`、`??` 等空安全操作符）防止空引用问题。

**【推荐】** 避免出现重复的代码（DRY 原则：Don't Repeat Yourself）。随意复制粘贴代码会导致修改时需要修改所有副本，容易遗漏。

### 日志规范

**【强制】** 应用程序中不可直接使用日志系统的 API，应依赖于日志框架的门面接口（如 `ILogger` / `ILogger<T>`），有利于维护和各个类的日志处理方式统一。

**【推荐】** 谨慎地记录日志。生产环境禁止输出 debug 日志；有选择地输出 info 日志。大量无效日志不利于系统性能提升和快速定位错误点。

**【强制】** 异常日志应包含两类信息：案发现场信息（参数、上下文）和异常堆栈信息。

**【参考】** 使用 warn 级别记录用户输入参数错误，error 级别只记录系统逻辑出错、异常等重要错误。

### 测试行为规范

**【强制】** 好的单元测试必须遵守 **AIR 原则**：
- **A**：Automatic（自动化）— 全自动执行，非交互式，必须用 Assert 验证而非人工检查
- **I**：Independent（独立性）— 用例之间不能互相调用，不能依赖执行先后次序
- **R**：Repeatable（可重复）— 不受外界环境影响（网络、服务、中间件等），通过 DI + Mock 隔离

**【强制】** 单元测试代码必须写在独立测试目录（如 `test/`），不允许写在业务代码目录下。

**【强制】** 核心业务、核心模块的增量代码确保单元测试通过。新增代码及时补充单元测试，如果新增代码影响了原有单元测试，请及时修正。

**【推荐】** 编写单元测试遵守 **BCDE 原则**：
- **B**：Border（边界值测试）— 循环边界、特殊取值、特殊时间点、数据顺序
- **C**：Correct（正确的输入）— 输入正确数据得到预期结果
- **D**：Design（与设计文档结合）— 根据设计文档编写单元测试
- **E**：Error（错误输入）— 强制错误信息输入并得到预期结果

**【推荐】** 单元测试语句覆盖率达到 70%；核心模块的语句覆盖率和分支覆盖率都要达到 100%。

**【推荐】** 和数据库相关的单元测试，可设定自动回滚机制，或对测试数据有明确的前后缀标识。

**【推荐】** 对于不可测的代码做必要的重构，使代码变得可测。建议在项目提测前完成单元测试，不建议项目发布后补充。

**【参考】** 业务代码避免：构造方法中做事过多、过多全局变量和静态方法、过多外部依赖、过多条件语句。

## 3. 代码规范（Code Conventions）

### 3.1 命名规范

#### 通用规则（所有语言）

**【强制】** 代码中的命名严禁使用拼音与英文混合的方式，更不允许直接使用中文。正确的英文拼写和语法可以让阅读者易于理解，避免歧义。

**【强制】** 杜绝完全不规范的缩写，避免望文不知义。如 `AbstractClass` 缩写成 `AbsClass`，`condition` 缩写成 `condi`。

**【推荐】** 任何自定义编程元素在命名时，使用尽量完整的单词组合来表达其意（代码自解释）。

**【推荐】** 如果模块、接口、类、方法使用了设计模式，在命名时体现出具体模式。如 `OrderFactory`、`LoginProxy`、`ResourceObserver`。

#### C# / .NET 专用规则

| 元素 | 规则 | 示例 |
| --- | --- | --- |
| 变量和参数 | Camel 命名规范（首字母小写） | `userName`, `fileManagerFactory` |
| 其他所有 | Pascal 命名规范（首字母大写） | `UserService`, `GetUserById` |
| 类的成员变量（private） | 前缀 `_` 开头 | `_userId`, `_member` |
| 常量 | 全大写，下划线分隔，const 修饰 | `VERSION_NUMBER`, `STATIC_FILE_NAME` |
| 接口 | 以 `I` 开头 | `IEnumerable`, `IMyService` |
| 抽象类 | 以 `Base` 或 `Provider` 结尾 | `RepositoryBase`, `DataProvider` |
| 异常类 | 以 `Exception` 结尾 | `UserNotFoundException` |
| 布尔值变量/属性/方法 | 以 `Is`、`Has`、`Can` 开头 | `isSystem`, `HasNamed`, `CanClick` |
| 异步方法 | 以 `Async` 结尾，返回 `Task` 或 `Task<T>` | `GetUserAsync(int id)` |
| 扩展方法类 | 以 `Extensions` 结尾 | `public static class FileNameExtensions` |
| 测试类/方法 | 类以 `Test` 结尾，方法以 `Test_` 开头 | `class UserTest` → `Test_UserName_Duplicate()` |
| 枚举声明 | 名词复数或单数形式，项用 Pascal | `enum MemberType { Unknown = 0 }` |

**【强制】** 参数名称不允许使用下划线、特殊字符、数字开头，也不能以下划线、特殊字符结尾。

**【强制】** 命名空间的每个级别使用 `.` 分隔，之间有且仅有一个自然语义的英语单词。

**【推荐】** 变量声明使用 `var` 关键字替代强制声明的数据类型，并务必对变量进行初始化。

**【强制】** 枚举项的每个值都必须显式赋值，且第一项必须是 `Unknown` 或 `None`，值为 0。

**【推荐】** 枚举声明不要加 `Enum` 后缀。

**【参考】** 分层命名：获取数据用 `Get/Find`，新增用 `Create/Add`，更新用 `Update`，删除用 `Delete/Remove`。

### 3.2 代码格式

**【强制】** 采用 4 个空格的缩进，禁止使用 tab 字符。

**【强制】** 大括号规范：
- 左大括号前换行、后换行
- 右大括号前换行、后换行
- 空大括号写成 `{}`

**【强制】** 左小括号和字符之间不出现空格，右小括号和字符之间不出现空格。

反例：`if( space == space )`

**【强制】** `if / for / while / switch / do` 等保留字与括号之间必须加空格。

**【强制】** 操作符的两端必须要有空格。

**【强制】** 注释的双斜线与注释内容之间有且仅有一个空格。`// 注释内容`

**【强制】** 单行字符数限制不超过 120 个，超过需要换行并遵守：
- 第二行相对第一行缩进 4 个空格，第三行起不再继续缩进
- 运算符与下文一起换行
- 方法调用的点符号与下文一起换行
- 方法的多个参数超过 5 个时，每个参数换行并与第一个参数对齐
- 参数逗号后换行（不在括号前换行）

**【强制】** 方法参数在定义和传入时，多个参数逗号后边必须加空格。`method("a", "b", "c");`

**【推荐】** 如果只有一句代码，尽量使用方法主体语句（表达式体）：

```cs
public Task<User> GetById(int id) => _context.GetAsync(id);
```

**【推荐】** 没有必要增加若干空格来使某一行的字符与上一行对应位置的字符对齐。

**【推荐】** 方法体内的执行语句组、定义语句组、不同的业务逻辑之间插入一个空行。相同业务逻辑和语义之间不需要插入空行。

**【推荐】** 类内方法定义顺序：公有方法或保护方法 > 私有方法 > 属性，并使用 `#region` 分类。

**【推荐】** 当一个类有多个构造方法或同名方法时，按顺序放置在一起，便于阅读。

**【推荐】** 使用可选参数代替更多的重载方法。`public void Add(int a, int b = 10, int c = 50)`

**【强制】** 异步方法最后一个参数是 `CancellationToken` 类型的可选参数：
```cs
public Task CreateAsync(User user, CancellationToken cancellationToken = default)
```

### 3.3 控制语句

**【强制】** 在控制语句中，即使只有一条语句，也不能省略大括号。

**【强制】** switch 块内每个 case 必须通过 `break/return` 等终止，或注释说明将继续执行到哪一个 case；switch 块必须包含一个 `default` 语句放在最后。

**【强制】** 避免超过 3 层的 if-else 嵌套。超过 3 层应使用卫语句（early return）、策略模式、状态模式等实现。

卫语句示例：
```cs
public void Today()
{
    if (IsBusy()) { Console.WriteLine("change time."); return; }
    if (IsFree()) { Console.WriteLine("go to travel."); return; }
    Console.WriteLine("stay at home.");
}
```

**【推荐】** 表达异常分支时少用 if-else 方式，改用卫语句。

**【推荐】** 不要在条件判断中执行复杂的语句，将复杂逻辑判断的结果赋值给一个有意义的布尔变量名以提高可读性：

```cs
var canCreate = (Directory.Exists("xxx") && File.Exists("xxx") || ...);
if (canCreate) { ... }
```

**【推荐】** 循环体中的操作要考量性能，定义对象、获取数据库连接、不必要的 try-catch 等尽量移至循环体外。

### 3.4 OOP 规范

**【推荐】** 引用类型的属性要在声明后给一个默认实例化，防止空引用异常：
```cs
public MyEntity Table { get; set; } = new MyEntity();
public IEnumerable<UserInfo> Users { get; set; } = new HashSet<UserInfo>();
```

**【强制】** 已经被广泛使用的接口如果要增加方法或修改参数，必须使用扩展方法来对原有方法进行扩展，防止所有调用处报异常。

**【强制】** 不要使用已过时的类或方法。调用方有义务考证过时方法的新实现。

**【强制】** Object 的 Equals 方法容易抛空指针异常，应使用常量或确定有值的对象来调用：`"test".Equals(object);`

**【强制】** 任何情况下禁止使用字符串拼接，特别是在循环内部。必须使用 `StringBuilder` 类或模板字符串/格式化字符串。

**【强制】** 类、类的成员与方法访问控制从严：
- 不允许外部直接 new 创建对象 → 构造方法 `private`
- 工具类 → `static` 修饰，尽量使用扩展方法
- 成员变量 → `private`，不允许共享给任何其他类
- 与子类共享的属性 → `protected`
- 仅本类使用的 `static` 成员 → `private`
- `static` 成员变量 → 考虑 `readonly`
- 仅类内部调用的方法 → `private`
- 仅对继承类公开的方法 → `protected`
- 慎用 `static` 属性，确保只提供只读功能

**【推荐】** 减少使用 `Tuple` 类，使用命名元组返回值以提供语义化字段名：
```cs
public (int age, string name) GetTuple() => (age: 1, name: "xyz");
var item = GetTuple();
Console.WriteLine(item.age);
```

### 3.5 并发处理

**【强制】** 获取单例对象需要保证线程安全，其中的方法也要保证线程安全。资源驱动类、工具类、单例工厂类都需要注意。

### 3.6 集合

**【强制】** 如果重写了 `Equals` 方法，就必须重写 `GetHashCode` 方法。

**【推荐】** 需要泛型集合类型进行值传递时，使用 `IEnumerable<T>` 类型而不是 `List<T>`。`List<T>` 有添加和移除功能，可能被调用方私自改写造成不可预期的错误。

**【推荐】** 在属性中对 `IEnumerable<T>` 初始化赋值时，使用 `HashSet<T>` 而不是 `List<T>`（效率更高）。

**【推荐】** 属性或返回值时，使用 `IReadOnlyList<T>` 作为泛型集合结果的返回类型（支持索引器访问，可用 for 循环）。

**【强制】** 集合判断是否有数据，使用 `System.Linq` 的 `Any()` 方法，不要用 `Count() > 0`。

**【推荐】** 使用 `System.Linq` 的扩展方法来对集合进行处理。

### 3.7 注释规范

**【强制】** 类、接口、属性、事件、委托、结构体、公开字段等，都必须使用 `/// <summary>...</summary>`（XML 文档注释）形式添加注释。不允许使用 `//` 对公共 API 进行说明。

**【强制】** 方法的签名中每一个参数都必须使用 `<param name="参数">注释</param>` 写注释。

**【强制】** 所有方法内部如果抛出异常，必须使用 `<exception cref="异常">抛出异常的原因</exception>` 在方法顶部标记。

**【强制】** 如果方法有返回值，必须使用 `<returns>返回值的说明</returns>` 说明。

完整 XML 注释示例：
```cs
/// <summary>
/// 创建指定名称的文件夹。
/// </summary>
/// <param name="name">要创建的文件夹名称。</param>
/// <exception cref="ArgumentNullException"><paramref name="name"/> 是 null。</exception>
/// <returns>创建成功返回 <c>true</c>；否则返回 <c>false</c>。</returns>
public bool MakeDir(string name)
{
    if (name == null) throw new ArgumentNullException(nameof(name));
    return false;
}
```

**【强制】** 所有的枚举类型字段必须要有注释，说明每个数据项的用途。

**【推荐】** 与其"半吊子"英文注释，不如用中文注释把问题说清楚。专有名词与关键字保持英文原文即可。

**【推荐】** 代码修改的同时，注释也要进行相应的修改，尤其是参数、返回值、异常、核心逻辑。

**【参考】** 谨慎注释掉代码。如果后续可能恢复，在上方详细说明原因并用 `/* */` 多行注释标记。如果无用则直接删除（代码仓库保存了历史）。

**【参考】** 特殊注释标记，请注明标记人与标记时间：
- **TODO**：(标记人，标记时间，[预计处理时间]) — 需要实现但目前未实现的功能
- **FIXME**：(标记人，标记时间，[预计处理时间]) — 代码有错误且不能工作，需要及时纠正

### 3.8 其他

**【推荐】** 任何数据结构的构造或初始化，都应指定大小，避免数据结构无限增长吃光内存。

**【推荐】** 及时清理不再使用的代码段或配置信息，避免程序过度臃肿。

### 3.9 项目结构与分层架构

#### 分层命名规范（.NET / C#）

| 对象类型 | 命名规则 | 示例 |
| --- | --- | --- |
| DO（Domain Object） | 名词复数，无前缀，与数据库表一一对应 | `Users`, `Companies` |
| DTO（Data Transfer Object） | 以 `Info` 作为后缀 | `UserInfo`, `CompanyInfo` |
| VO（View Object） | 以 `ViewModel` 作为后缀，格式：模块名称 + 操作分类 + ViewModel | `UserCreateViewModel`, `MembershipDisplayListViewModel` |
| SO（Specification Object） | 以 `Spec` 作为后缀，超过 3 个参数封装 | `UserListSpec`, `AuthoritySpec` |

#### 命名空间分层

**【强制】** 项目命名空间每一层的名字作为第三级，最多不超过 5 级。

推荐分层结构：
- **Presentation**（表现层）：Local / Web / App
- **Exposes**（暴露层）：SDK / WebApi / 服务
- **Applications**（应用层）
- **Domain**（领域层）：DomainService / DomainModel / DomainEntity / DomainEvent
- 传输对象命名空间：ViewModels / Specifications / DataTransferObjects / DataObjects

#### 第三方库规范

**【强制】** 暴露对外的 API 一定要提供被调用方重写的功能，使客户端可以根据需求改写内部调用算法。

**【强制】** 版本号命名方式：主版本号.次版本号.修订号（SemVer 语义化版本）
- 主版本号：产品方向改变、大规模 API 不兼容、架构不兼容升级
- 次版本号：保持相对兼容性，增加主要功能特性
- 修订号：保持完全兼容性，修复 Bug、新增次要功能
- 起始版本号必须为 1.0.0（非 0.0.1）

**【强制】** 依赖于一个库群时，必须定义一个统一的版本变量，避免版本号不一致。

**【强制】** 库的发布必须使用 Release 方式，包中必须带有 XML 注释文件。
