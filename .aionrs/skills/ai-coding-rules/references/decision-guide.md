# Decision Guide — Handling Edge Cases

当代码规范遇到实际开发的复杂场景时，如何做出正确的判断。

---

## 规范约束力分级

规则按约束力分为三级，与 DotNetSpec 一致：

| 级别 | 含义 | 违反后果 |
| --- | --- | --- |
| **【强制】** | 必须遵守，代码评审时一票否决 | 可能导致系统故障、安全漏洞、严重性能问题 |
| **【推荐】** | 应当遵守，特殊情况下可破例 | 可能降低可维护性、可读性，但不会导致故障 |
| **【参考】** | 建议遵守，由开发者自行判断 | 影响代码风格一致性 |

---

## 何时破例

### 可接受的破例（仅对【推荐】和【参考】级别）

| 场景 | 可接受的做法 | 条件 |
| --- | --- | --- |
| 紧急 Bug 修复 | 跳过测试先行，直接修 | 修复后 24h 内补测试 |
| 第三方库限制（.NET） | 使用 `Tuple` 而非命名元组 | 目标框架不支持 C# 7.0+ |
| 性能关键路径 | 函数超过 50 行 | 注释解释为何不能拆分 + 性能基准 |
| 遗留代码兼容 | 不强制新规范 | 在文件头部标记 `@legacy` |

### 不可破例（【强制】级别无条件遵守）

| 场景 | 不可接受 | 替代方案 |
| --- | --- | --- |
| 跳过参数校验 | 信任用户输入 | 至少做类型/长度/范围校验 |
| 硬编码密钥 | 把密钥写在代码里 | 使用环境变量或 Secret 管理 |
| 删他人代码不沟通 | 直接删除不清楚用途的代码 | 先注释 + TODO，确认后删除 |
| 忽略所有错误 | `catch { }` 空处理 | 至少记录日志 + 用户提示 |
| 字符串拼接循环 | `str += item` 循环拼接 | StringBuilder 或模板字符串 |
| 修改已发布接口 | 直接改接口签名 | 扩展方法 + 新接口版本 |
| finally 中 return | `finally { return x; }` | 在 try 块外 return |
| 生产环境 debug 日志 | 大量输出 Debug 级别日志 | 降为 Trace 或条件编译 |
| 捕获常规检查异常 | `catch(NullReferenceException)` | 用 `if(obj == null)` 防御 |

## 修改范围决策树

当不清楚一个修改属于什么范围时，按以下流程判断：

```
是否只影响一个函数内部？
├── 是 → strict（直接改）
└── 否 → 是否只影响当前文件？
    ├── 是 → file（检查文件内其他引用点）
    └── 否 → 是否跨文件但同模块？
        ├── 是 → module（告知影响范围后改）
        └── 否 → 是否跨模块？
            ├── 是 → project（必须确认后执行）
            └── 边界不清 → 拆分修改，逐个确认
```

### .NET 场景的模块范围判断

- 同一 Controller + Service + Repository 链 = 同一模块
- 跨 Bounded Context（如 User 模块 → Order 模块）= project 级别
- 修改 Domain Entity + 对应 Migration = project 级别（影响数据库结构）
- 修改 Startup.cs / Program.cs = project 级别
- 修改 appsettings.json = project 级别

## 冲突处理优先级

当多个规范冲突时，按以下优先级处理：

1. **安全性** > 所有其他规则（安全规则不可妥协）
2. **用户明确指令** > 任何默认规则
3. **项目既有规则** > 本 skill 默认规则
4. **代码范围规则** > 行为规范 > 代码规范
5. **可逆性** — 容易回滚的操作优先于不可逆操作
6. **【强制】** > **【推荐】** > **【参考】**

---

## 不确定时的行为

当遇到以下情况，**停下来询问用户**，不要猜测：

1. 不清楚某段代码的用途和影响范围
2. 修改可能影响 API 契约（公共接口签名变更）
3. 修改涉及并发/竞态条件
4. 修改涉及事务边界或数据一致性
5. 两个方案都可以实现目标且无法判断优劣
6. 被要求修改标记为「禁止修改」的代码区域
7. 需要违反【强制】级别的规范（必须先确认）
8. 不清楚项目使用的是哪种分层架构（DDD / 三层 / Clean Architecture）

---

## 批量修改决策

当修改需要跨越多个文件（≥3 个文件）：

1. 先列出所有受影响的文件清单
2. 描述每个文件的变更摘要
3. 说明变更间的依赖关系
4. 获得确认后再逐个文件执行
5. 每完成一个文件报告进度和结果

### .NET 项目批量修改特别注意

- 修改 Entity 时检查是否影响 Migration、DTO、ViewModel
- 修改接口时检查所有实现类和调用方
- 修改 appsettings.json 时检查 `IOptions<T>` 绑定类
- 新增 NuGet 包时统一版本变量管理

## 代码评审检查清单

AI 生成的代码在提交前应自查以下项目：

### 【强制】级别检查

- [ ] 命名未被缩写，未使用拼音或中文
- [ ] 私有成员变量前缀 `_`（C#）
- [ ] 接口以 `I` 开头（C#）
- [ ] 异步方法以 `Async` 结尾（C#）
- [ ] 常量全大写+下划线分隔
- [ ] 缩进 4 空格（非 Tab）
- [ ] 单行不超过 120 字符
- [ ] 控制语句不省略大括号
- [ ] switch 有 default 分支
- [ ] if-else 嵌套不超过 3 层
- [ ] 无字符串拼接循环
- [ ] 无硬编码密钥/密码
- [ ] 异常不用于流程控制
- [ ] catch 不空处理
- [ ] 无已过时 API 调用
- [ ] Equals 重写时也重写 GetHashCode

### 【推荐】级别检查

- [ ] 布尔变量使用 Is/Has/Can 前缀
- [ ] 单语句方法使用表达式体
- [ ] 集合暴露使用 IEnumerable<T> 而非 List<T>
- [ ] 集合初始化使用 HashSet<T>
- [ ] 集合非空判断使用 Any()
- [ ] 复杂条件提取为命名布尔变量
- [ ] 引用类型属性有默认实例化
- [ ] 方法参数不超过 5 个（超过用规约对象）
- [ ] 单元测试覆盖核心分支
